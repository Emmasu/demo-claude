Introduction

This short intro sets up the core idea of the entire course. The instructor, Elie Schoppik (Head of Technical Education at Anthropic), explains what Skills are and why they matter in the age of AI agents.

"Skills are folders of instructions that extend your agent's capabilities with specialized knowledge."

What You'll Learn in This Course

  • What Skills are and why they exist
  • How to create reusable Skills using the open standard format
  • How to build custom Skills for code generation, code review, data analysis, and research
  • How to use Skills with Claude AI, the Claude API, Claude Code, and the Claude Agent SDK

The Core Idea

Every Skill contains a SKILL.md file — a Markdown file with instructions, plus supporting files like scripts, documentation, and assets. When an agent encounters a task that matches a skill's description, it automatically loads and uses that skill.

The format is an open standard. Skills you build for Claude also work on Gemini CLI, Codex, and other skills-compatible platforms.

💡
Key mental model Think of a Skill as a reusable "how-to guide" for your agent. Instead of explaining the same workflow every time, you package it once and the agent knows how to do it automatically.

Why Use Skills — Part I

Watch Video 11 min

This lesson demonstrates the problem Skills solve, using a real marketing campaign analysis workflow as a concrete example.

The Problem: Repeated Prompts

Without Skills, every time you want Claude to perform a complex task, you must paste the full instructions into the chat. For example, analyzing marketing campaigns requires explaining:

  • What data quality checks to perform
  • How to calculate ROI, cost per acquisition, net profit
  • The specific budget reallocation rules
  • Any business rules or benchmarks

This is wasteful — you're consuming the context window with instructions that never change, and you have to repeat it in every new conversation.

"If you find yourself typing the same prompt across conversations, you should consider transforming that into a skill."

The Solution: Package It as a Skill

A Skill packages that workflow into a folder you upload once. Claude automatically recognizes when to use it based on the skill description, and loads the instructions on demand.

Anatomy of a SKILL.md File

Every Skill is built around a SKILL.md file. Here's the structure:

analyzing-marketing-campaigns/
  ├── SKILL.md       # required — main instructions
  └── references/
      ├── benchmarks.md  # industry benchmarks
      └── rules.md       # reallocation rules

The SKILL.md file itself looks like this:

markdown / SKILL.md---
name: analyzing-marketing-campaigns
description: Analyzes marketing campaign performance data.
            Use when asked to analyze campaigns, calculate ROI,
            or reallocate marketing budgets.
---

# Analyzing Marketing Campaigns

## Step 1: Data Quality & Funnel Analysis
- Check total record count and missing values
- Flag anomalies in impressions, clicks, conversions
- Compare click-through rate vs benchmark (2%)
- Compare conversion rate vs benchmark (5%)

## Step 2: Efficiency Metrics
- Calculate Return on Ad Spend (ROAS = Revenue / Cost)
- Calculate Cost Per Acquisition (CPA = Cost / Conversions)
- Calculate Net Profit (Revenue - Cost)

## Step 3: Budget Reallocation
- Refer to references/rules.md for reallocation logic
- Apply rules per channel
- Output recommended budget by channel

YAML Frontmatter Rules

FieldRequiredRules
nameYesLowercase letters, numbers, hyphens. No spaces. No reserved words (Claude, Anthropic).
descriptionYesExplains what the skill does AND when to use it. Include keywords that trigger activation.

Naming Convention

Follow the verb + ing pattern:

  • analyzing-marketing-campaigns
  • generating-practice-questions
  • reviewing-cli-commands
  • creating-brand-presentations

Why Use Skills — Part II

Watch Video 8 min

Part II digs deeper into how Skills work technically, focusing on context window efficiency and what makes them different from just pasting text into a prompt.

Skills Are More Than Text

Skills aren't just Markdown documents linking to other documents. They are functional components that can include:

  • Executable scripts (Python, Bash)
  • Referenced resources (PDFs, CSVs, images)
  • Assets (templates, logos, schemas)

When Claude runs a skill, it can actually execute code, read files, and produce structured outputs — not just follow text instructions.

Progressive Disclosure — Protecting the Context Window

One of the most important technical aspects of Skills is progressive disclosure. Here's how it works:

Agent starts → Only skill name + description in context
↓ user request matches skill description ↓
SKILL.md loads → Full instructions now in context
↓ task requires additional files ↓
References / Scripts load → Only what's needed

This is critical at scale. If you have 20 skills installed, you don't want all 20 sets of instructions consuming tokens. Only the relevant skill's details are loaded when needed.

ℹ️
Tool definitions are always in context; Skills are not Tool definitions (like bash, filesystem) are always present in the context window. Skills, by contrast, use progressive disclosure — only the name/description stays resident unless the skill is invoked.

Best Use Cases for Skills

Skills shine in situations where Claude lacks company-specific or domain-specific knowledge:

Use CaseWhy a Skill Helps
Brand guidelinesClaude doesn't know your company's fonts, colors, logo rules
Financial analysis workflowsYour firm has specific formulas, thresholds, report formats
Legal review processesJurisdiction-specific rules, internal approval checklists
Newsletter designLayout conventions, tone of voice, section structure
Code review standardsProject-specific conventions, security rules, patterns

The Philosophy: Simpler Agents + Composable Skills

The old approach was to build many specialized agents (a coding agent, a finance agent, a research agent). The new approach:

"Rather than building domain-specific agents, use simpler agents with common infrastructure (bash, filesystem) enhanced through composable skills."

One general-purpose agent + a library of skills = the functionality of many specialized agents, with far less complexity.

Skills vs Tools, MCP, and Subagents

Watch Video 7 min

This lesson maps out the full landscape of components in a modern agentic system and clarifies when to use each one.

The Four Components

🔧
Tools

Low-level capabilities: Bash, filesystem read/write, web search, API calls. Always in context.

📋
Skills

High-level knowledge: step-by-step workflows, domain expertise, reusable procedures. Loaded on demand.

🔌
MCP

Model Context Protocol: connects agents to external systems (databases, Google Drive, Notion, Slack).

🤖
Subagents

Specialized agents spawned by a main agent. Each has its own context, tools, and permissions.

The Analogy

"Think of tools as a hammer and a saw and some nails. Skills are higher-level knowledge — like knowing how to build a bookshelf."

Tools give the agent physical capabilities. Skills give it the knowledge of what to build and how to build it well.

How They Differ

ComponentLives InPurposeUse When
Tools Always in context Execute actions (run code, read files, call APIs) Agent needs to do something
Skills Loaded progressively Provide procedural knowledge and workflows Agent needs to know how to do something
MCP External servers Connect to external data and systems Agent needs to access external resources
Subagents Separate context windows Parallel, isolated, specialized task execution Task can be parallelized or needs isolation

Real Example: Customer Insight Analyzer

The lesson walks through a "Customer Insight Analyzer" to show how all four components work together:

Main Agent uses MCP → CRM Database
↓ spawns subagents ↓
Subagent A
Interview Analysis Subagent B
Survey Analysis
↓ both use ↓
Skill: categorizing-feedback Skill: summarizing-findings
  • Main agent orchestrates everything and accesses the database via MCP
  • Subagents run the analysis in parallel (faster, isolated contexts)
  • Skills ensure each subagent follows the same predictable workflow
  • Tools (Bash, WebSearch) provide execution capabilities to each agent

Decision Guide: When to Use What

SituationUse
Repeatable, predictable workflowSkill
Full agentic logic with decisionsSubagent
Access to external data/servicesMCP
Execute code, read files, call APIsTool
Parallel execution of independent tasksSubagents
Context-window protection for many skillsProgressive Disclosure

Creating Custom Skills

Watch Video 16 min

This is the core technical lesson on how to write good skills from scratch. It covers structure, best practices, two real examples, and how to evaluate your skills.

Full Skill Directory Structure

your-skill-name/
  ├── SKILL.md         # required — YAML frontmatter + instructions
  ├── scripts/          # optional — executable code
      ├── step1.py
      └── step2.py
  ├── references/       # optional — extra documentation
      ├── guidelines.md
      └── examples.md
  └── assets/           # optional — templates, images, data
      ├── template.xlsx
      └── logo.png

SKILL.md Best Practices

Frontmatter

  • name: lowercase, hyphens, verb+ing format (e.g., generating-practice-questions)
  • description: explain both what it does and when to use it — include trigger keywords

Body Content

  • Keep under 500 lines total — reference external files for anything longer
  • Use forward slashes for all paths (not backslashes) for cross-platform compatibility
  • Write clear, sequential steps — number them explicitly
  • Define edge cases and explain when steps can be skipped
  • Balance specificity vs flexibility based on your use case

Freedom vs. Constraint

Low Freedom (Specify Exactly)High Freedom (Let Claude Decide)
Script execution orderColor palette choices
File naming conventionsWriting style and tone
Output file structureLayout and visual design
Required validation checksExample selection

Example 1: Generating Practice Questions

This skill generates educational questions from lecture notes, supporting multiple output formats. 

markdown / SKILL.md---
name: generating-practice-questions
description: Generates practice questions from lecture notes or
            educational content. Use when creating study materials,
            quizzes, or exam prep content.
---

# Generating Practice Questions

## Required Input
- Lecture notes or educational content (text, PDF, or markdown)
- Desired output format (Markdown, LaTeX, or PDF)
- Difficulty level (beginner / intermediate / advanced)

## Question Progression
Generate questions in this order:
1. True/False — test basic comprehension
2. Multiple Choice — test understanding with distractors
3. Short Answer — require explanation in 2-3 sentences
4. Coding Problems — apply concepts programmatically
5. Application — connect to real-world scenarios

## Output Formats
- Markdown: use assets/markdown_template.md
- LaTeX: use assets/latex_template.tex (compile with pdflatex)
- PDF: generate via LaTeX pipeline

## Edge Cases
- If content is under 500 words, skip coding and application sections
- If no difficulty specified, default to intermediate
generating-practice-questions/
  ├── SKILL.md
  └── assets/
      ├── markdown_template.md
      └── latex_template.tex

Example 2: Analyzing Time Series Data

This skill runs a deterministic sequence of Python scripts to diagnose CSV data before forecasting. 

markdown / SKILL.md---
name: analyzing-time-series-data
description: Diagnoses time series CSV data characteristics before
            forecasting. Use when analyzing trends, seasonality,
            or preparing data for prediction models.
---

# Analyzing Time Series Data

## Dependencies
Required Python libraries: pandas, numpy, matplotlib, statsmodels, scipy

## Workflow (execute in order)
1. Run scripts/01_load_validate.py — check format, missing values, frequency
2. Run scripts/02_visualize.py — plot raw series, rolling mean/std
3. Run scripts/03_autocorrelation.py — ACF and PACF plots
4. Run scripts/04_decompose.py — trend, seasonality, residuals
5. Run scripts/05_summary.py — write diagnosis to output/summary.md

## Outputs
- output/summary.md — written diagnosis with recommendations
- output/plots/ — all generated visualizations
- Each script logs errors to output/errors.log

## Optional Steps
- If user requests stationarity test: run scripts/06_adf_test.py
- If user requests forecast: proceed to forecasting skill after diagnosis

Testing Your Skills

Treat skills like software — unit test them:

Test TypeWhat to Check
Input handlingDoes it work with various input formats? What happens with edge cases?
Workflow executionDo all steps run in the right order? Are dependencies handled?
Output formatDoes the output match the expected structure and file locations?
LaTeX compilationDo LaTeX outputs actually compile to PDF without errors?
Human reviewGather feedback across multiple Claude model versions

Evaluating with Skill-Creator

You can run the skill-creator tool against your skill to score it on best practices. The two example skills scored:

  • generating-practice-questions: 9/10 (minor conciseness improvements suggested)
  • analyzing-time-series-data: 10/10 (excellent on duplication, frontmatter quality, conciseness)

Skills with the Claude API

Watch Video 17 min

When building applications with the Anthropic Messages API directly, skills don't auto-configure. This lesson shows how to wire everything up manually.

🚨
Critical difference Skills you create in Claude AI and Claude Desktop are NOT shared with the Claude API or Claude Code. You must set up code execution and filesystem access manually.

Two Components You Must Add

1. Code Execution Tool

Provides a sandboxed container where Claude can run Bash commands, Python scripts, and manage files.

  • Pre-installed libraries available (Python, common packages)
  • No internet connection (can't pip install arbitrary packages)
  • RAM, disk, and CPU are constrained
  • Enabled via beta headers in the API request

2. Files API

Handles uploading and downloading files between your application and the sandboxed execution environment.

Full Workflow

pythonimport anthropic

client = anthropic.Anthropic()

# Step 1: Upload your skill folder via Files API
def upload_skill(skill_path: str) -> str:
    with open(skill_path, "rb") as f:
        response = client.beta.files.upload(
            file=(skill_path, f, "text/markdown"),
        )
    return response.id

# Step 2: Upload your input data file
with open("lecture_notes.pdf", "rb") as f:
    input_file = client.beta.files.upload(
        file=("lecture_notes.pdf", f, "application/pdf")
    )

skill_file_id = upload_skill("generating-practice-questions/SKILL.md")

# Step 3: Send message to Claude with skill + code execution
response = client.beta.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=4096,
    betas=["files-api-2025-04-14", "code-execution-2025-05-22"],
    tools=[{"type": "code_execution"}],
    container={
        "type": "persistent",
        "files": [skill_file_id, input_file.id]
    },
    messages=[{
        "role": "user",
        "content": "Generate practice questions from the lecture notes."
    }]
)

# Step 4: Extract output file IDs from response and download
for block in response.content:
    if hasattr(block, "file_id"):
        content = client.beta.files.download(block.file_id)
        with open("output_questions.md", "wb") as f:
            f.write(content)

# Step 5: Clean up (optional)
client.beta.files.delete(skill_file_id)

What Happens Inside the Container

When Claude receives the request, it uses progressive disclosure — it reads the SKILL.md file first, then loads additional assets only when needed. Claude will:

1
Read SKILL.md

Claude reads the instructions to understand the workflow

2
Execute code steps

Runs Python/Bash scripts inside the sandbox per the skill instructions

3
Load assets as needed

Reads templates, references only when the step requires them

4
Write output files

Produces result files with IDs you can download via the Files API

Skills Demonstrated in This Lesson

  • generating-practice-questions — from lecture notes (LaTeX/PDF output)
  • analyzing-time-series-data — from CSV files with visualizations
  • Both combined with the built-in docx skill to also produce Word document output
⚠️
No internet in the sandbox The container cannot make HTTP requests or install packages at runtime. Make sure all required libraries are available in the pre-installed environment or bundle them with your skill.

Project: CLI Task Manager (Python + Typer)

The demo project is a command-line task manager built with Python and the Typer framework. Its architecture:

task-manager/
  ├── commands/     # each CLI command is a separate file
      ├── add.py
      ├── list.py

  ├── models.py    # data models
  ├── storage.py   # persistence logic
  ├── display.py   # terminal output formatting
  ├── CLAUDE.md    # project context for Claude
  └── .claude/
      └── skills/
          ├── adding-cli-commands/
          ├── generating-cli-tests/

CLAUDE.md — Project Context File

CLAUDE.md lives in the project root and maintains context across Claude Code conversations. It documents:

  • Technology stack (Python 3.11, Typer, pytest)
  • Architecture patterns (one file per command, registration in __init__.py)
  • Coding conventions (type annotations required, use Rich for display)
  • Key file locations and what they do

Without CLAUDE.md, Claude would need to re-explore the codebase in every conversation.

The Three Skills

Skill 1: adding-cli-commands

markdown / SKILL.md---
name: adding-cli-commands
description: Creates new CLI commands following project conventions.
            Use when adding new commands to the task manager.
---

# Adding CLI Commands

## Steps
1. Create commands/{command_name}.py with:
   - Typer app function with proper type annotations
   - Docstring explaining the command
   - Import from models.py and storage.py
   - Use display.py for all terminal output
2. Register in commands/__init__.py
3. Add entry to CLAUDE.md command list

## Required Patterns
- Use `typer.Option` for optional flags with `--flag-name` format
- Use `typer.Argument` for required positional args
- Always validate input before storage operations
- Return appropriate exit codes

Skill 2: generating-cli-tests

markdown / SKILL.md---
name: generating-cli-tests
description: Generates pytest tests for CLI commands. Use when
            writing tests for any task manager command.
---

# Generating CLI Tests

## Test Structure
1. Create tests/test_{command_name}.py
2. Use pytest fixtures from conftest.py:
   - `runner` — CliRunner instance
   - `tmp_storage` — isolated storage for tests
3. Test categories (in order):
   - Happy path: normal expected input
   - Edge cases: empty input, boundary values
   - Error cases: invalid input, missing required args
   - Flag combinations: multiple options together

## Requirements
- Each test function has a single assertion focus
- Use descriptive test names: test_add_creates_task_with_title()
- Mock external I/O (file writes) in unit tests
- Integration tests use tmp_storage fixture

Skill 3: reviewing-cli-commands

markdown / SKILL.md---
name: reviewing-cli-commands
description: Reviews CLI command implementations for quality,
            security, and best practices. Use to validate any
            new or modified command before merging.
---

# Reviewing CLI Commands

## Checklist
1. Type annotations on all function params and return types
2. Input validation before any storage/IO operations
3. Error messages are user-friendly (not stack traces)
4. No hardcoded file paths (use config/constants)
5. Follows naming conventions from CLAUDE.md
6. Registered correctly in commands/__init__.py
7. No security issues (no eval, no shell injection via user input)

Using Subagents in Claude Code

Rather than reviewing and testing in the main context, the lesson introduces specialized subagents: 

python (Claude Code agent definition)# code-reviewer subagent
# Limited to read-only tools — can only inspect, not modify
code_reviewer = Agent(
    description="Reviews code quality, security, and convention compliance",
    tools=["Bash", "Glob", "Grep", "Read"],
    skills=["reviewing-cli-commands"]
)

# test-generator-runner subagent
# Has write access to create test files and run them
test_agent = Agent(
    description="Generates pytest tests and executes them",
    tools=["Bash", "Glob", "Grep", "Read", "Edit", "Write"],
    skills=["generating-cli-tests"]
)
🚨
Subagents do NOT inherit skills Skills are not passed down from parent to subagent automatically. You must explicitly list the skills each subagent needs. If you forget, the subagent won't follow your conventions.

Example Workflow: Adding the "edit" Command

1
Generate the command

Main agent uses the adding-cli-commands skill to create commands/edit.py with all required patterns

2
Dispatch code-reviewer subagent

Subagent checks the new file against the review checklist without touching the main context

3
Dispatch test-generator-runner subagent

Subagent writes tests per the testing skill, runs them with pytest, reports results

4
Main agent reviews results

If both subagents pass, the feature is done. If not, iterate.

The learning-a-tool Skill

markdown / SKILL.md---
name: learning-a-tool
description: Creates comprehensive learning guides for open-source
            tools and libraries. Use when asked to learn or
            research a new tool, library, or framework.
---

# Learning a Tool

## Research Phase
Dispatch three specialized subagents in parallel:
1. **Documentation Researcher** — official docs, API reference, quickstart guides
2. **Repository Analyzer** — GitHub README, examples/, issues, recent commits
3. **Web Researcher** — blog posts, tutorials, Stack Overflow, Reddit discussions

## Learning Path Structure
Organize output into progressive difficulty levels:

**Level 0: Overview & Motivation**
- What problem does this tool solve?
- When should/shouldn't you use it?
- How does it compare to alternatives?

**Level 1: Installation & Setup**
- Installation commands
- Configuration required
- Hello world example

**Level 2: Core Concepts**
- Key abstractions and their relationships
- Mental model for how it works

**Level 3: Practical Patterns**
- Common use cases with code examples
- Best practices and gotchas

**Level 4: Advanced Topics**
- Performance optimization
- Edge cases
- Production considerations

## Output Format
Generate a folder structure:
  output/{tool-name}/
    README.md        — overview and navigation
    resources.md     — all links found
    learning-path/
      00-overview.md
      01-setup.md
      02-core-concepts.md
      03-patterns.md
      04-advanced.md
    examples/
      hello-world.{ext}
      common-patterns.{ext}

Full Agent Implementation

python / agent.pyimport asyncio
from claude_agent_sdk import Agent, AgentSettings
from dotenv import load_dotenv

load_dotenv()

# Subagent: searches official documentation
doc_researcher = Agent(
    description="Researches official documentation, API references, and quickstart guides",
    prompt="Search for official documentation and guides for the tool provided.",
    tools=["WebSearch", "WebFetch"],
)

# Subagent: analyzes the GitHub repository
repo_analyzer = Agent(
    description="Analyzes GitHub repositories for examples, patterns, and issues",
    prompt="Find and analyze the GitHub repository for the tool provided.",
    tools=["Bash", "WebSearch", "WebFetch", "Read"],
)

# Subagent: gathers community knowledge
web_researcher = Agent(
    description="Gathers community knowledge: tutorials, blog posts, Stack Overflow",
    prompt="Find community resources, tutorials, and discussions about the tool.",
    tools=["WebSearch", "WebFetch"],
)

# Main orchestrator
main_agent = Agent(
    description="Research orchestrator that creates comprehensive learning guides",
    tools=["Write", "Bash", "WebSearch", "WebFetch", "Skill"],
    subagents=[doc_researcher, repo_analyzer, web_researcher],
    settings=AgentSettings(
        setting_sources=["user", "project"]  # loads .claude/skills/
    ),
    mcp_servers=[{
        "type": "notion",
        "auth_token": "..."
    }]
)

async def main():
    result = await main_agent.run(
        "Create a comprehensive learning guide for MinerU (a PDF extraction library)"
    )
    print(result)

asyncio.run(main())

Tool Configuration Per Subagent

🚨
Always be explicit about tools and skills for subagents Every subagent needs its own explicit tool list. Subagents don't inherit anything from the parent — not tools, not skills, not MCP connections.

Security Considerations for Production

The instructor flags important limitations of the current implementation:

  • No user confirmation before executing Write or Bash commands
  • No interrupt mechanism (unlike Claude Code's UI)
  • No permission-based tool execution framework

For production systems, you need to add confirmation prompts before destructive operations.

The course wraps up by summarizing the layered architecture you've learned:

  • Skills are reusable, portable, and open-standard — build once, use everywhere
  • MCP connects agents to the external world
  • Subagents enable parallel, specialized, isolated execution
  • These three components compose together to build powerful agentic systems

The key insight from the whole course: you don't need dozens of specialized agents. A general-purpose agent with well-designed composable skills, the right tools, and MCP access can handle complex real-world workflows reliably and predictably.

Skill Writing Checklist

  • YAML frontmatter with name and description
  • Name follows verb-ing-noun pattern, lowercase, hyphens
  • Description explains both what AND when
  • Under 500 lines (use references/ for overflow)
  • Forward slashes for all paths
  • Sequential numbered steps
  • Edge cases documented
  • Scripts in scripts/, docs in references/, files in assets/
  • Tested like software (happy path, edge cases, output format)

Key Gotchas

🚨
Subagents don't inherit anything from the parent Always explicitly declare tools, skills, and MCP servers for each subagent.
🚨
Claude AI skills ≠ API skills Skills uploaded to Claude AI are not available in the Messages API. You must upload them programmatically via the Files API.
⚠️
API sandbox has no internet The code execution container cannot install packages or make HTTP requests. Pre-bundle dependencies.
💡
Skills are an open standard The same SKILL.md format works on Claude, Gemini CLI, Codex, and any other skills-compatible platform.
